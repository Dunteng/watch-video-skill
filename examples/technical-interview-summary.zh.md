## 核心概念拆解

### 短期记忆

- **定义**：当前上下文窗口里可直接被模型看到的信息。
- **边界**：速度快，但容量有限，不能当成长期知识库。
- **例子**：最近几轮对话、当前工具调用结果、正在修的 bug 状态。
- **为什么重要**：决定当前回答是否连贯，适合保存即时任务状态。
- **视频证据**：`[00:00:55]` 处字幕提到“当前上下文窗口”和“最近对话”。

### 工作记忆

- **定义**：长任务中临时保留的结构化草稿或任务状态。
- **边界**：生命周期通常跟任务绑定，不一定跨会话长期保留。
- **例子**：代码任务中记录改了哪些文件、下一步要验证什么、哪个问题尚未修复。
- **为什么重要**：避免长任务中丢失中间决策。
- **视频证据**：`[00:01:16]` 处字幕提到“任务里的临时草稿区”。

### 长期记忆

- **定义**：跨会话可复用的稳定知识和偏好。
- **边界**：不能把所有对话塞进去；需要筛选、压缩、更新和删除。
- **例子**：用户明确偏好、项目规则、历史决策、常见错误。
- **为什么重要**：让 Agent 下次能动态召回稳定上下文。
- **视频证据**：`[00:01:48]` 处字幕提到“高价值记忆”和“用户偏好、验证过的重要决策”。

## 常见误区

### 误区：记忆系统就是向量数据库

- **为什么错**：向量库只解决一部分检索问题，不解决写入时机、记忆质量、遗忘策略和上下文预算。
- **正确说法**：面试回答应覆盖写入、检索、压缩、淘汰和权限边界。
- **证据**：`[00:00:31]` 处字幕明确说“不能只回答接数据库/向量库”。

### 误区：长期记忆越多越好

- **为什么错**：低质量记忆会污染检索结果，占用上下文，甚至覆盖最新用户意图。
- **正确说法**：长期记忆应保存稳定、高价值、已验证的信息。
- **证据**：`[00:02:02]` 处字幕提到“真正应该保存的是高价值记忆”。

## 可执行方法

- **前提**：任务需要跨轮次或跨会话保持上下文。
- **步骤**：
  1. 把信息分为短期、工作、长期三类。
  2. 给长期记忆设置写入门槛：稳定、可复用、已确认。
  3. 检索时按当前任务目标过滤，而不是无差别注入。
  4. 上下文快满时压缩中间过程，只保留结论、证据和下一步。
  5. 定期清理过期偏好、错误决策和一次性任务细节。
- **检查清单**：
  - 是否说明了写入策略？
  - 是否说明了检索触发条件？
  - 是否说明了遗忘或更新机制？
  - 是否区分 memory 和 context engineering？
- **适用场景**：代码助手、研究助手、长期项目 Agent、客服 Agent。
- **不适用场景**：一次性问答、无需跨轮上下文的小任务。

## 面试回答模板

### 30 秒版

我会把 Agent 记忆分成短期上下文、任务级工作记忆和长期记忆。短期上下文服务当前推理，工作记忆服务长任务推进，长期记忆只保存稳定、高价值、可复用的信息，比如用户偏好、项目规则和历史决策。重点不是存得多，而是设计何时写入、何时检索、何时压缩和何时遗忘。

### 2 分钟版

我不会把记忆系统简单等同于向量数据库。向量库只是检索层，完整设计还要包括写入策略、检索策略、上下文注入、压缩和淘汰。比如用户偏好和项目规则可以进入长期记忆，当前 bug 的排查状态更适合工作记忆，最近对话则留在短期上下文。检索时要根据当前任务目标选择相关记忆，避免把无关历史塞进上下文。上下文快满时，应该压缩工具调用和中间过程，只保留结论、证据、失败经验和下一步。

### 追问应对

- 如果问“怎么防止记忆污染”：设置写入门槛、来源标记、更新时间、置信度和删除机制。
- 如果问“用不用向量库”：可以用，但它只是检索实现，不是记忆系统本身。
- 如果问“怎么评估”：看任务完成率、错误复现率、过期记忆命中率和用户纠错次数。

### 不推荐回答

“接一个向量数据库，把所有聊天记录存进去，需要时检索出来。”

这个回答缺少写入策略、遗忘策略、上下文预算和证据边界。
